# Cascade Mobile — Roadmap

Foundational integration work, in order:

1. ✅ **`src/lib/` foundation** — `api.ts` (base URL config), `query-client.ts`, `trpc.ts` (typed client against `AppRouter`), and a cookie/secure-store layer for session persistence.
2. ✅ **Auth flow** — login screen (`fetch` → cookie capture), session bootstrap on launch, `auth.me`, logout, and routing between authed/unauthed states. Lives in `src/lib/auth/` + `src/app/login.tsx` + the root-layout auth gate.
3. ✅ **Org context** — `OrgProvider`/`useOrg` (`src/lib/org-context.tsx`) registers `setOrgContextGetter` from a synchronous module ref, derives the effective org from `auth.me` (member → own org; superadmin → persisted choice clamped to `availableOrgs`, else default), persists the superadmin's choice via `expo-secure-store` (`src/lib/org-storage.ts` + `.web.ts` → `localStorage`), and invalidates React Query on switch. A minimal superadmin-only switcher lives in `src/components/org-switcher.tsx` (members render nothing); the provider is mounted in the root layout.
4. ✅ **First feature screen** — projects list → project detail / runs, proving the end-to-end typed path. Originally a top-level `src/app/projects/` route group; **relocated under `(tabs)/projects/` in step 5**. Data flows through thin org-scoped hooks (`src/hooks/use-projects.ts`, `src/hooks/use-project-runs.ts`) that wrap `trpc.projects.list` / `trpc.runs.list` and gate on `useOrg().isReady`. Shared presentational pieces live in `src/components/run-status-badge.tsx` (status→color pill, graceful unknown-status fallback) and `src/components/query-states.tsx` (`Loading`/`EmptyState`/`ErrorState`), with pull-to-refresh on the `FlatList`.
5. ✅ **Native bottom tabs IA** — the placeholder Home/Explore tabs are replaced by a real `NativeTabs` bar (`src/components/app-tabs.tsx`, web parity in `app-tabs.web.tsx`) of up to four tabs, each owning its own nested `Stack` under `src/app/(tabs)/<name>/`: **Runs** (default), **Projects** (relocated here; detail reshaped into a sections list → `[projectId]/[section]` placeholders), **Settings** (account info + active org + relocated sign-out, plus a Users placeholder), and a superadmin-only **Global** hub (Global Runs / Webhook Logs / Agent Definitions / Organizations placeholders). Tab icons use SF Symbols (iOS) + Material Symbols (Android) via the `NativeTabs.Trigger.Icon` `sf`/`md` props — no PNG assets. The Global tab is gated on `useIsSuperadmin()` (`src/lib/auth/use-is-superadmin.ts`, reading `auth.me` `role` — distinct from `useOrg().isSuperadmin`, which tracks org count) and its stack self-guards with a `Redirect`. The org switcher moved from the old Home tab into a header-right control (`src/components/org-switcher-header.tsx`, a `Modal` reusing `OrgSwitcher`) wired into every tab stack via a shared `stackScreenOptions`. The auth gate redirects authenticated users to `/runs` (named tab dirs mean `/` has no leaf). Runs/Settings/Global ship as navigation-ready IA placeholders; their feature content is deferred to follow-up cards.

6. ✅ **Runs feed** — the Runs tab's placeholder is replaced by a mobile-native, cross-project card feed (not the web's multi-column table). Data flows through a new org-scoped infinite hook (`src/hooks/use-runs.ts`) that wraps `trpc.runs.list` across all projects (no `projectId`), gates on `useOrg().isReady`, and paginates by offset/limit via the bare `trpcClient` (now exported from `src/lib/trpc.ts`) — folding optional `{ status, agentType, projectId }` filters into the query key so a filter change resets paging. Each run renders as a `RunCard` (`src/components/run-card.tsx`) with a scannable hierarchy (agent type + status badge → project / work item → relative-time · duration · cost · iterations meta → PR link), reusing the `projects/index.tsx` `FlatList` + `RefreshControl` + `query-states` idiom with infinite scroll + a footer spinner. Running runs show a self-ticking elapsed time via `LiveDuration` (`src/components/live-duration.tsx`); pure formatting helpers live in `src/lib/relative-time.ts` (`formatRelativeTime` / `formatDuration` / `formatCost` / `formatAgentType`). `RunStatusBadge` gained an amber `timed_out` status. `showOrg` is gated on the account role (`useIsSuperadmin`). The mobile filter UI is a Modal bottom sheet (`src/components/runs-filter-sheet.tsx`) of reusable single-select `FilterChips` (`src/components/filter-chips.tsx`) for Status / Agent type / Project, opened from a header-right "Filter (n)" trigger composed alongside the org switcher.

7. ✅ **Run detail header** — the run-detail placeholder (`src/app/(tabs)/runs/[runId].tsx`) is replaced by a real header screen. A thin `useRun(runId)` hook (`src/hooks/use-run.ts`) wraps `trpc.runs.getById`, gated on `useOrg().isReady` and a truthy `runId` (same guard as `useProjectRuns`). The screen sets the native header title to the formatted agent type via `formatAgentType`, renders a `RunStatusBadge` status pill, and conditionally shows Work Item and PR links using the existing `ExternalLink` component (in-app browser on native). Loading, error, and not-found states use the shared `query-states.tsx` components. Field names follow the `RunListItem` narrow-type idiom from `run-card.tsx`. Foundation for the run-detail epic (section nav + sections in follow-up cards #20–#24).

8. ✅ **Run detail — LLM Calls section** — the `llm-calls` section placeholder on the run detail screen (`src/app/(tabs)/runs/[runId].tsx`) is replaced by a real expandable-list component (`src/components/run-llm-calls.tsx`). Each row shows a collapsed model · token summary · cost line and expands on tap to reveal input/output/cached tokens, duration, cost, model, tool call names, and text/thinking previews. Data flows through `useRunLlmCalls(runId)` (`src/hooks/use-run-llm-calls.ts`), mirroring `useRun` (gated on `useOrg().isReady` and a truthy `runId`). Two new pure helpers in `src/lib/relative-time.ts`: `formatTokens` (comma-grouped, regex-based — no `Intl`) and `formatLlmCost` (sub-cent-aware USD formatter). The section nav wiring was a single branch change — the `llm-calls` key/label already existed in `run-section-tabs.tsx`.

9. ✅ **Run detail — Overview section** — the `overview` section placeholder on the run detail screen (`src/app/(tabs)/runs/[runId].tsx`) is replaced by a real `RunOverview` component (`src/components/run-overview.tsx`) that renders the run's key fields as full-width label/value rows (iOS-Settings detail style). Fields: engine, trigger type, model, iterations (with max), gadget calls, started/completed (relative time), duration (live-ticking via `LiveDuration` for running runs, stored `formatDuration` for completed), result (explicit boolean — `Succeeded`/`Failed`), cost (`formatLlmCost`, sub-cent aware), error (styled in error-red `#e5484d`), and output summary (inline for short text, wrapped in `Collapsible` when > 160 chars or >= 3 newlines). The component takes the already-fetched run as a prop (no new hook/query — `runs.getById` via `useRun` carries every field). The `RunDetail` type was extended to intersect `RunOverviewData` with the header-specific fields. Empty/null fields are auto-skipped. Reuses `LiveDuration`, `formatDuration`, `formatLlmCost`, `formatRelativeTime`, `Collapsible`, and the `DetailRow` label/value idiom from `run-llm-calls.tsx`.

10. ✅ **Run detail — Logs section** — the `logs` section placeholder on the run detail screen (`src/app/(tabs)/runs/[runId].tsx`) is replaced by a presentational `RunLogs` component (`src/components/run-logs.tsx`). The component owns a two-segment Cascade Log / Engine Log sub-toggle (reusing the `Pressable` + `ThemedView` + `ThemedText` strip idiom from `RunSectionTabs`) and renders the active log in monospaced text (`ThemedText type="code"`) inside a vertical `ScrollView` with word-wrap. Per-type empty states show when a log is `null` / empty. Data follows Plan A: `cascadeLog` / `engineLog` fields are read from the already-loaded `runs.getById` response (extended on the `RunDetail` narrow type) — no extra hook or request. The component stays purely presentational (the screen passes the data as props). Log streaming/tailing, search/filter, download, and virtualization are explicitly out of scope.

11. ✅ **Project Work section — runs feed** — the "Work" section placeholder under the project detail screen (`src/app/(tabs)/projects/[projectId]/work.tsx`) is replaced by a real project-scoped runs feed. The screen mirrors the Runs tab feed (`runs/index.tsx`) — `FlatList` + `RunCard` + `RefreshControl` + infinite scroll + footer spinner + `query-states` — but omits the filter sheet/trigger and passes `{ projectId }` to the existing `useRuns` hook (`src/hooks/use-runs.ts`), which already folds `projectId` into both the query input and key. The project detail sections list (`src/app/(tabs)/projects/[projectId]/index.tsx`) now explicitly routes the "Work" row to `/projects/[projectId]/work` (type-safe, unambiguous) rather than relying on the generic `[section]` dynamic route. The header title defaults to "Work" (overridable via a `label` param from the sections list); `headerRight` is inherited from `ProjectsLayout`'s `stackScreenOptions` (org switcher). Pressing a run card pushes `/runs/[runId]` in the Runs tab stack — the same cross-tab navigation pattern established in item 7.

11. ✅ **Project Work section — filter controls** — the Work feed now has Status and Agent-type filter controls, reusing `RunsFilterSheet` + `FilterChips` with the Project dimension hidden (`hideProjectFilter` prop) and `projectId` pinned to the route value. `activeFilterCount` gained an `{ includeProject }` option so the badge and "Clear all" don't permanently light up when the project is always set. The header-right composes the "Filter (n)" trigger alongside `OrgSwitcherHeader` (overriding `headerRight` replaces the layout's default, so the org switcher is re-included explicitly). The empty state reflects whether filters are active. All changes are backward-compatible — the Runs tab's existing usage is untouched.

Keep this list current as work lands.
